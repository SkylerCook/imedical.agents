# 打印 i18n 案例复盘：6095804

本文记录 6095804「门诊医生-总览打印中部分打印单据不支持多语言」的处理经验。

本文件是案例复盘，不是通用规范。文中所有可复用内容均按“候选经验”处理；只有在后续多个打印 i18n 需求中反复验证后，才适合提升为 plugin rule、skill 或脚本默认行为。

本文不记录服务器地址、账号、密码、token、namespace、远程路径、患者样本等敏感信息。

## 本次案例事实

- 需求现象：门诊医生总览打印中，部分单据预览或打印时仍显示源语言文案。
- 本次重点排查过的范围：总览打印入口、后端返回打印数据、XML 打印模板记录、EN 模板创建与命中。
- 本次确认过的一个链路形态是：入口页触发打印，后端返回包含模板相关字段和打印数据的结果，前端或后端再使用模板标识渲染 XML 打印模板。
- 本次只覆盖了部分总览打印路径，不代表所有打印都走同一套链路。
- 后端返回打印数据不绝对是 JSON，也可能是 string、分隔符拼接、HTML/XML 片段、类 JSON 字符串或其他结构。
- 预览仍显示源语言时，曾出现过根因只是后端代码未编译，而不是模板选择逻辑仍错误。

## 不要泛化的点

- 不是所有打印都会走 `GetXMLTemplateId(PrintTemp)`。
- 不是所有打印都会返回 `PrintTemp`。
- 不是所有打印都使用 XML 打印模板。
- 不是所有预览和打印共用同一模板解析路径。
- 不是所有后端打印数据都是 JSON。
- 不是所有中文残留都应该在当前工程改；当前工程外接口返回值由对应业务组负责。
- 本次没有完成所有打印单据的字段级改造，不能把本次结果包装成完整通用工作流。

## 候选经验：链路定位

后续处理打印 i18n 需求时，优先做事实定位，不要预设链路。

建议按以下顺序确认：

1. 从入口页或按钮事件定位实际调用的后端方法。
2. 抓取实际返回的打印数据，先判断数据形态。
3. 查找是否存在模板字段，例如 `PrintTemp`、`PreviewXMLName`、`templateId` 或其他项目特有模板标识。
4. 判断模板渲染路径：XML 模板、HTML/CSP、字符串直出、第三方打印接口或其他方式。
5. 再判断中文残留位于模板、后端固定文案、字典展示值、业务输入，还是外部接口返回值。

不要用某个历史需求的链路替代当前需求的实际链路。

## 候选经验：XML 打印模板

本次曾验证 XML 模板存在语言版本时，模板命中是打印多语言的一部分。但该经验只适用于实际链路确认为 XML 模板渲染的打印。

候选做法：

- 从实际打印数据或配置中发现模板标识。
- 导出源模板记录，确认模板流字段内容。
- 只翻译用户可见 XML `defaultvalue`，不改坐标、变量名、条码、打印机配置等结构信息。
- 新建 `{Template}-{LANG}` 语言模板时，写入语言代码和原始模板标识。
- 写入后必须验证目标语言模板记录、XML 可解析、源语言残留和模板命中。

注意：`i18n-xml-print-template-sync` 只适合能定位到 XML 模板记录的场景，不应作为所有打印 i18n 的默认入口。

## 候选经验：后端打印数据分类

后端打印数据中的用户可见文本，需要先分类再改造。

| 类型 | 判断标准 | 候选处理 |
|---|---|---|
| 固定文案 | 代码中硬编码的标题、页码、标签、金额单位、括号标识等 | 使用页面级翻译 helper，带变量时用整句占位符 |
| 字典展示值 | 从 Global、SQL、持久类字段、字典表字段直接取出的原文展示值 | 使用公共字典翻译门面，例如 `DHCDoc.Common.Translate.GetTransXxx` |
| 业务输入 | 备注、用户手工录入说明、病人或医生临时输入内容 | 默认不翻译 |
| 外部接口返回 | 当前工程无法看到源码或归属其他业务组的接口返回值 | 当前工程不二次翻译，转交对应业务组 |
| 未确认来源 | 调用链能看到方法名但无法确定底层数据来源 | 继续追链路；追不到时标记待确认，不猜测改造 |

## 候选经验：字典翻译位置

字典翻译应贴近原始字段来源，不应在最终变量上无脑套 `GetTransXxx`。

例如，当代码明确从 `^ARCIM(...,1)` 取出 `ARCIMDesc` 时，可以在该取值后立即调用医嘱项字典翻译门面。若后续变量又被外部接口返回值覆盖，则不应对覆盖后的最终变量继续套同一个字典翻译方法。

这种写法能避免把不同来源、不同业务含义的文本误认为同一个字典字段。

## 候选经验：编码约束

本工程约定：

- 后端 `.cls` 编码为 UTF-8。
- 前端文件编码为 GB2312。

后端新增中文翻译 key 时，按 UTF-8 校验。前端文件如需改造，应按前端 GB2312 规则处理，避免因为工具默认编码导致乱码。

## 多 Agent 协作候选模式

后续类似需求可以拆给多个 Agent，但每个 Agent 必须有明确边界。

- Explorer Agent：定位入口、后端方法、实际返回数据形态和模板链路。
- Template Agent：仅在确认存在 XML 模板链路时处理模板导出、翻译、写入和验证。
- Classifier Agent：基于实际打印数据标注字段来源和改造方式，不改代码。
- Coding Agent：只按已确认字段分类做最小代码改造。
- Verification Agent：基于编译后的实际打印返回数据验证模板命中、源语言残留和 fallback 行为。

协作原则：先事实定位，再分类，再改造，再验证。不要用推断替代运行后事实。

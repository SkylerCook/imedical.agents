# iMedicalXC 医生站集成范围分析

判断第三方系统集成需求的团队归属。

## 场景：范围分析（功能归属分析）

**输入**：第三方集成需求、厂商接口规格或工作流嵌入请求
**输出**：团队归属决策及理由、交接文档（如适用）

### 分析标准

- 是否需要嵌入医生站工作流（医嘱、诊断、处方）？
- 是否需要在临床决策过程中实时交互？
- 第三方接口是否需要 XML/JSON 格式转换？
- 是否存在需要组装的复杂数据结构？
- 第三方响应是否影响业务逻辑或需要用户确认？
- 是否仅为无工作流影响的数据抽取？
- 是否使用浏览器不兼容技术（DLL/OCX/ActiveX）？
- 第三方的流程意图是什么？（参见 `imedicalxc-doctor-extend-architecture` → `references/frontend-integration-patterns.md` §模式3）
  - 通知型：第三方自行处理 UI 交互 → 评估是否可作为异步任务委托给平台组
  - 交互型：第三方返回数据供 HIS 展示/交互 → 属于医生站组，需要同步等待
- **需求是否涉及用户登录/登出、会话管理、Token 认证或权限控制？** → 这些属于基础平台能力，不是医生站范围。即使集成嵌入在医生-facing 页面，底层机制仍归属基础平台组。

### 决策规则

1. **工作流集成规则**：嵌入临床工作流 → 医生站组
2. **数据抽取规则**：纯数据抽取且无工作流影响 → 医院信息平台组
3. **技术兼容性规则**：DLL/OCX/ActiveX → 使用 WebSysAddins 平台
4. **基础平台规则**：登录/登出、会话管理、Token 认证、权限控制、系统级配置 → 基础平台组

### 输出格式

```
归属判断: [医生站组 / 医院信息平台组 / 基础平台组]
置信度: [高/中/低]
理由:
1. [工作流影响分析]
2. [数据抽取分析]
3. [技术要求分析]
4. [基础平台能力分析 — 如适用]
```

## 详细范围分析规则

### 医生站组职责

| 集成类型 | 职责 | 示例 |
|-----------------|---------------|---------|
| **工作流嵌入** | 实时嵌入临床工作流 | 医嘱录入时 SPD 库存校验 |
| **交互型集成** | 需要用户交互/确认 | 医保资格校验弹窗 |
| **业务逻辑集成** | 第三方响应影响业务决策 | 外部系统药品相互作用检查 |
| **中间件型集成** | 通过 WebSysAddins 中间件（DLL/OCX/ActiveX） | 读卡器、指纹仪 |

### 医院信息平台组职责

| 集成类型 | 职责 | 示例 |
|-----------------|---------------|---------|
| **纯数据抽取** | 无工作流影响的异步数据推送 | 每日处方数据上传监管平台 |
| **事件触发数据推送** | 事件触发、非阻塞 | 患者入院时向第三方推送数据 |
| **定时数据同步** | 定时批量数据抽取 | 每小时患者数同步 |
| **数据转换** | 无业务逻辑的格式转换 | HL7/FHIR → 自定义 XML/JSON |

**平台组关键特征**：
- 数据抽取不改变业务流程
- 不阻塞前端操作
- 不校验第三方响应的业务逻辑
- 使用现有 HIS 标准接口
- 作为异步后台任务执行

### 基础平台组职责

基础平台组负责 HIS 系统的公共基础能力，这些能力被所有业务模块（门诊、住院、患者管理、药房等）共用，不属于任何一个业务域。

| 能力类型 | 职责 | 示例 |
|-----------------|---------------|---------|
| **用户登录/登出** | 用户身份认证、登录态建立与销毁 | 第三方系统接入 HIS 单点登录（SSO） |
| **会话管理** | Session 创建、续期、销毁、超时控制 | 第三方需要验证 HIS 会话有效性 |
| **Token 认证** | Access-Token 生成、校验、刷新 | 第三方通过 Token 访问 HIS API 的认证机制 |
| **权限控制** | 角色管理、菜单权限、功能权限、数据权限 | 第三方用户需要映射 HIS 角色与权限 |
| **系统配置** | 系统级参数、开关、字典元数据管理 | 第三方功能的系统级开关与参数配置 |
| **HISUI 框架** | 前端基础框架、全局组件、主题样式 | 第三方界面嵌入时的框架兼容性 |

**基础平台组关键特征**：
- 能力被全系统共用，不局限于某个业务模块
- 修改影响面广，需全量回归测试
- 通常涉及安全合规要求（如等保、密码法）
- 变更需要平台组架构评审
- 登录/登出相关的任何需求，无论是否在医生站页面触发，均归属基础平台组

### 决策矩阵

| 需求 | 负责团队 | 理由 |
|-------------|-----------------|--------|
| 医嘱录入时 SPD 库存校验 | 医生站组 | 嵌入工作流，影响医嘱提交 |
| 医保资格校验弹窗 | 医生站组 | 需要用户确认，实时阻塞 |
| 外部药品相互作用检查 | 医生站组 | 影响临床决策，需要校验 |
| 每日处方数据上传 | 医院信息平台组 | 纯数据抽取，异步，无工作流影响 |
| 患者入院数据推送 | 医院信息平台组 | 事件触发，异步，非阻塞 |
| 读卡器（DLL） | 医生站组 + 中间件组 | 需要 WebSysAddins 实现 Chrome 兼容 |
| 指纹仪（OCX） | 医生站组 + 中间件组 | 需要 WebSysAddins 实现 Chrome 兼容 |
| 第三方 SSO 单点登录接入 | 基础平台组 | 涉及登录认证机制，全系统共用 |
| 第三方获取当前登录用户信息 | 基础平台组 | 涉及会话管理，需评估 Token 传递方案 |
| 第三方验证 HIS 会话有效性 | 基础平台组 | 会话校验属于基础平台能力 |
| HIS 登录页面嵌入第三方认证 | 基础平台组 | 登录流程改造，影响全系统 |
| 登出时通知第三方系统 | 基础平台组 | 登出生命周期回调，属于会话管理 |

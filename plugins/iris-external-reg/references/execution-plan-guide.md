# 执行计划文档编写规范

## 使用方式

当需要从接口规范文档生成开发计划、修订接口清单或补充接口详细设计时读取本文。本文给出执行计划应包含的结构、模式 A / 模式 B 的判断规则、待确认项写法和可复制模板。

## 目录

- 文档位置
- 执行计划文档结构
- 数据格式说明
- 接口清单
- 公共数据分析
- 接口详细设计
- 状态跟踪
- 模板文件
- 接口列表模板文件

## 概述

执行计划文档是根据接口规范文档解析后生成的开发指导文档，用于明确接口清单、公共数据抽取、接口详细设计等开发所需的完整信息。

## 文档位置

执行计划文档统一存放于 `docs/output/iris-interface/<文档名>/` 目录下，文件命名格式：

```text
<接口规范文档名>-执行计划.md
<接口规范文档名>-接口列表.json
```

## 执行计划文档结构

### 一、数据格式说明

#### 1.1 传输格式

| 类别 | 格式 | 说明 |
|------|------|------|
| HTTP Header | Key-Value | 认证信息、Content-Type等 |
| 请求体 | JSON/XML | 统一使用UTF-8编码 |
| 响应体 | JSON/XML | 统一使用UTF-8编码 |

#### 1.2 内部调用格式（核心层交互）

| 类别 | 格式 | 说明 |
|------|------|------|
| HIS→平台推送 | JSON/XML + HTTP POST | 通过sendINFData推送至省平台 |
| 平台→HIS查询 | JSON/XML + HTTP POST | 接收平台请求，返回响应数据 |
| 内部核心调用 | XML + DHCExternalService | 翻译层与核心服务交互 |

#### 1.3 核心服务调用模式

```text
1. 构建Entity请求对象 → ##class(DHCExternalService.RegInterface.Entity.XXX).%New()
2. 设置请求属性 → s input.XXX = value
3. 导出XML字符串 → d input.XMLExportToString(.xmlStr, "Request")
4. 调用核心方法 → ##class(DHCExternalService.RegInterface.XXXMethods).Method(xmlStr)
5. 解析XML响应 → 检查 ResultCode，遍历结果集合，转换为对应数据格式输出
```

#### 1.4 视图/数据源

| 视图/数据源 | 用途 | 确认状态 |
|-------------|------|---------|
| ^CTLOC | 科室基础信息 | ✅ 已确认 |
| ^CTPCP(doctorId) | 医生基础信息 | ✅ 已确认 |
| ^RB("RES") | 排班资源 | ✅ 已确认 |
| ^RBAS(resId, subId) | 排班子表（号源） | ✅ 已确认 |
| ^User.PAAR | 患者档案 | ✅ 已确认 |
| ^OEORD(ordId) | 医嘱/挂号记录 | ⏳ **待确认** — 需人工核实字段含义和取片位置 |

**⚠️ 数据源确认约束**：
- 对于标记为「待确认」的数据源（如 `^OEORD`），必须在执行计划阶段形成 TODO 列表
- 禁止在字段含义未确认的情况下，直接推断取片位置并编写代码

**⚠️ 多院区数据隔离约束**：
- 所有接口必须接收 `hospitalCode` 入参，并转换为 `hospId`
- 遍历 `^CTLOC` 时必须使用 `^CTLOC(0, "Hosp", hospId, locId)` 按院区索引
- 遍历 `^RB("RES")` 时，必须通过科室ID校验院区归属（`^CTLOC(deptId)` 第 22 片）
- 执行计划文档中必须标注每个接口的院区过滤策略（参见接口详细设计中的「多院区处理」章节）
- 执行计划文档中必须包含「待确认项」章节，列出所有需要人工确认的数据源和字段

### 四、接口详细设计

为每个接口编写详细设计，包含以下部分：

#### 4.1 基本信息

- **请求格式**：JSON/XML POST / 数据格式说明
- **架构模式**：模式A（推送/上报） / 模式B（查询/操作）
- **调用核心层**：Global遍历 / XML互操作 + DHCExternalService方法
- **Entity类**：对应的Entity类名（模式B必填，模式A不填）
- **TradeCode**：交易代码（模式B必填，模式A不填）
- **核心方法**：调用的 `DHCExternalService` 方法（模式B必填，模式A不填）
- **数据来源**：HIS全局变量名（模式A必填，模式B不填）

#### 4.2 参数表

**输入参数**：

| 参数名 | 类型 | 必填 | 说明 | 校验规则 |

| hospitalCode | String | 是 | 医院编码 | 非空校验 |

**输出参数**：

| 参数名 | 类型 | 说明 |
|--------|------|------|
| orderId | String | 订单ID |

#### 4.3 处理流程

描述接口的处理步骤：

1. 入参解析 + hospitalCode转换
2. 构建XML请求对象
3. 调用核心方法
4. 解析响应返回

#### registerOrder - 挂号

**请求格式：** JSON/XML POST
**架构模式：** 模式B（查询/操作）
**调用核心层：** XML/DHCExternalService
**Entity类：** RegisterOrderRq
**TradeCode：** 5003
**核心方法：** ##class(DHCExternalService.RegInterface.SelfRegMethods).RegisterOrder()

**多院区处理：**
- `hospitalCode` 入参必填，转换为 `hospId` 后传入 `input.HospitalId`
- 号源有效性校验时，需通过 `^RB("RES", resId)` 获取科室ID，再校验科室是否属于目标院区

**输入参数：**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| hospitalCode | String | 是 | 医院编码 |
| patientId | String | 是 | 患者ID |
| scheduleId | String | 是 | 排班ID |

**输出参数：**

| 参数名 | 类型 | 说明 |
|--------|------|------|
| orderId | String | 订单ID |
| orderNo | String | 订单号 |

**处理流程：**
1. 入参解析 + hospitalCode转换为hospId
2. 构建XML请求对象（模式B：入参包含完整业务数据），设置 `input.HospitalId = hospId`
3. 调用核心方法
4. 解析响应返回

**错误场景：**

| 错误码 | 场景 | 提示信息 |
|--------|------|----------|
| -1 | 参数缺失 | 缺少必填参数hospitalCode |
| -2 | 号源不可用 | 该号源已被锁定 |
| -3 | 跨院区访问 | 号源不属于目标院区 |

#### pushDepartmentSingle - 单条科室数据推送

**请求格式：** JSON POST
**架构模式：** 模式A（推送/上报）
**调用核心层：** Global遍历
**数据来源：** ^CTLOC

**多院区处理：**
- `hospitalCode` 入参必填，转换为 `hospId`
- 从 `^CTLOC(0, "Hosp", hospId, locId)` 按院区索引读取科室，禁止直接遍历 `^CTLOC(locId)`
- 校验 `req.code` 对应的科室是否属于目标院区

**输入参数：**

| 参数名 | 类型 | 必填 | 说明 | 备注 |
|--------|------|------|------|------|
| hospitalCode | String | 是 | 医院编码 | 用于校验和推送 |
| code | String | 是 | 科室编码 | 查询标识，用于从HIS读取完整数据 |

**输出参数：**

| 参数名 | 类型 | 说明 |
|--------|------|------|
| code | Integer | 0=成功，-1=失败 |
| msg | String | 提示信息 |

**处理流程：**
1. 入参解析（仅校验标识参数）
2. 从 `^CTLOC(0, "Hosp", hospId, locId)` 按院区索引读取科室完整数据
3. 校验 `req.code` 对应的科室是否属于目标院区
4. 构建推送数据（使用HIS实际数据，禁止透传入参中的业务字段）
5. 调用 `sendINFData` 推送至省平台

**关键约束：**
- 入参禁止包含 name, address, description 等业务字段
- 所有业务数据必须从 `^CTLOC` 读取
- 必须按院区过滤，禁止跨院区推送数据

**待确认项（如适用）：**

| TODO-ID | 待确认内容 | 影响范围 | 确认状态 |
|---------|-----------|---------|---------|
| TODO-A1 | `^CTLOC` 第 X 片是否为 Y 字段 | 字段取值 | ⏳ 待确认 |

**错误场景：**

| 错误码 | 场景 | 提示信息 |
|--------|------|----------|
| -1 | 参数缺失 | 缺少必填参数hospitalCode |
| -2 | 数据不存在 | 科室编码不存在 |
| -3 | 跨院区访问 | 科室不属于目标院区 |
| -4 | 推送失败 | 省平台返回错误 |

#### reportAppointment - 预约数据上报

**请求格式：** JSON POST
**架构模式：** 模式A（推送/上报）
**调用核心层：** Global遍历
**数据来源：** `^OEORD`（⏳ **待确认** — 需人工核实是否为预约数据的正确数据源，以及具体字段含义）

**多院区处理：**
- `hospitalCode` 入参必填，转换为 `hospId`
- 遍历 `^OEORD` 时，需通过医嘱关联的就诊记录（`^OEORD(oeordId)` 第 2 片为 AdmID）获取院区信息，或确认 `^OEORD` 是否有院区索引
- 仅统计属于目标院区的预约数据

**输入参数：**

| 参数名 | 类型 | 必填 | 说明 | 备注 |
|--------|------|------|------|------|
| hospitalCode | String | 是 | 医院编码 | 用于校验和推送 |
| startDate | String | 否 | 开始日期 | 默认当天 |
| endDate | String | 否 | 结束日期 | 默认当天 |

**输出参数：**

| 参数名 | 类型 | 说明 |
|--------|------|------|
| code | Integer | 0=成功，-1=失败 |
| msg | String | 提示信息 |
| data.successCount | Integer | 上报成功条数 |

**处理流程：**
1. 入参解析（仅校验标识参数）
2. 从 `^OEORD` 全局变量读取预约数据（⏳ **待确认**：具体取片位置、状态码枚举值）
3. 按院区过滤：通过医嘱关联的就诊记录获取院区ID，仅保留目标院区数据（⏳ **待确认**：具体过滤方式）
4. 按排班维度聚合统计各状态数量
5. 构建推送数据（使用HIS实际数据）
6. 调用 `sendINFData` 推送至省平台

**关键约束：**
- 入参禁止包含完整业务字段
- 所有业务数据必须从 HIS 全局变量读取
- **禁止自行臆测 `^OEORD` 字段含义**，必须在 TODO 确认后编码
- 必须按院区过滤，禁止跨院区上报数据

**待确认项：**

| TODO-ID | 待确认内容 | 影响范围 | 确认状态 |
|---------|-----------|---------|---------|
| TODO-R1 | `^OEORD` 是否为预约数据的正确数据源（vs `^RBAS(..., "APPT")`） | 数据源选择 | ⏳ 待确认 |
| TODO-R2 | `^OEORD` 第 9 片是否为 resId，第 10 片是否为 subId | 排班关联 | ⏳ 待确认 |
| TODO-R3 | `^OEORD` 第 13 片是否为预约日期 | 日期过滤 | ⏳ 待确认 |
| TODO-R4 | `^OEORD` 第 15 片状态码的完整枚举值及含义（A/N/C/D 等） | 状态统计 | ⏳ 待确认 |
| TODO-R5 | `^OEORD` 与 `^RBAS(..., "APPT")` 的数据关系 | 数据一致性 | ⏳ 待确认 |
| TODO-R6 | `^OEORD` 是否包含院区标识，或如何通过关联表获取院区信息 | 院区过滤 | ⏳ 待确认 |
| TODO-R7 | 多院区环境下 `^OEORD` 的遍历方式（是否有院区索引） | 性能优化 | ⏳ 待确认 |

**错误场景：**

| 错误码 | 场景 | 提示信息 |
|--------|------|----------|
| -1 | 参数缺失 | 缺少必填参数hospitalCode |
| -2 | 数据不存在 | 指定日期范围内无预约数据 |
| -3 | 跨院区访问 | 预约数据不属于目标院区 |
| -4 | 推送失败 | 省平台返回错误 |

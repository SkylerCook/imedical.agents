# 模式 A：性能诊断分析

## 前置判断：是否有明确分析目标

### 场景 A：用户已指定目标

用户提供了 traceId / 接口名 / 方法名 → 直接进入第一步。

### 场景 B：用户未指定目标（如"帮我看看有什么慢接口"）

1. **询问时间范围**：使用 `AskUserQuestion` 询问用户查询时间范围；用户不指定则默认最近 3 天
2. **搜索慢接口**：参考 graylog-search.md 使用 MCP 工具搜索 `full_message:"接口响应超过15"`，用 `fields="*"` 获取 traceId 等完整字段
3. **获取精确统计**：用 `aggregate_logs` 按接口分组获取精确次数和耗时统计
4. **计算慢请求占比**（关键步骤，每接口 2 次 MCP 调用）：

   > **已知限制**：`full_message` 字段为 keyword 类型，不支持 `AND` 组合查询（如 `full_message:"接口响应超过15" AND full_message:"orderPay"` 返回空），需用以下两步法。

   **第①步 — 发现 URL 路径**：
   - `search_logs_relative`，query 为 `full_message:{方法名}`，limit=3，fields=message
   - 从返回的 HeaderFilter 日志中提取 URL 路径，格式：`HeaderFilter.traceId.header.set,url=http://x.x.x.x:port/路径/方法名,traceId=...`
   - 如果同方法名对应多个 URL（如 ipcare/opcare 两套），各取一条

   **第②步 — 统计总调用量（分母）**：
   - `aggregate_logs`，query 为 `full_message:"/{提取的URL路径}"`
   - 如有多条 URL，分别查询后求和
   - **返回仅一个数字**（`total_matched`），无论总调用量多大 token 消耗恒定

   **已知限制**：
   - `aggregate_logs` 的 `fetchLimit` 默认 5000（最大 10000），若单接口 10h 调用量超此值会截断，此时用 `full_message:{方法名}` 的 `total_results` 作为上界估算
   - 定时任务类接口（批量处理、报表导出等）占比高不等于代码缺陷，需在报告中单独标注

   计算：`慢请求占比 = 慢调用次数 / 总调用次数 × 100%`
5. **汇总列表**：按接口分组，列出每个接口的慢调用次数、总调用次数、慢请求占比、风险判定
   
   | 占比 | 风险判定 |
   |------|---------|
   | > 50% | 系统性风险 — 几乎所有调用都慢 |
   | 10% ~ 50% | 高频风险 — 显著比例受影响 |
   | 1% ~ 10% | 需关注 — 偶发但不可忽视 |
   | < 1% | 个例 — 可暂缓处理 |

6. **询问用户意图**（使用 `AskUserQuestion`）：
   - "输出报告" — 生成汇总报告（仅整理归类，不分析代码）
   - "挑一个具体分析" — 用户选择一个接口进入第一步
   - **禁止自动分析所有慢接口**

## 工作流程

```
前置判断 → 1. 收集信息 → 2. 读取代码 → 3. 日志分析 → 4. 精确排名 → 5. 输出
```

## 第一步：收集信息

从用户输入中提取：
- **接口全限定名**：如 `com.mediway.his.ipcare.oeord.controller.IpCareOeOrdItemPortalController#unUseMulti`
- **traceId**：如 `8869509280484950536`
- **Graylog 环境**：使用 `AskUserQuestion` 询问用户选择环境（参考 graylog-search.md 的环境列表）
- **输出目录**：默认输出到当前项目目录，用户可另行指定。文件名为 `{方法名}性能分析报告.md`

## 第二步：读取代码

### 2.1 定位 Controller

搜索 Controller 类，记录：接口路径 (`@RequestMapping` + `@PostMapping`)、方法签名和入参 DTO、调用的 BLH 实例。

### 2.2 追踪 BLH 继承链

```
ext/BLH (注解实现) → Abstract (业务层) → comoe Abstract (公共库) → ...
```

每层记录：文件路径、关键方法逻辑、外部调用（Feign/Mapper/Service）、是否有循环处理。

### 2.3 识别潜在 N+1 点

检查以下模式：
- `for` 循环内调用 Mapper / Feign
- `for` 循环内单条 INSERT / UPDATE
- 同一方法多次调用同参数

一旦发现 N+1，必须追踪到**循环入口**和**循环内的调用行**，记录精确的文件名:行号。

### 2.4 检查 CT/CF 表缓存使用

**规则**：`ct_`（码表）和 `cf_`（配置表）开头的表，数据变动频率低、数据量小，**分析是否适合使用本地缓存**。适合但未使用的需标出。

检查步骤：
1. 识别代码中涉及的 Mapper 操作，筛选表名为 `ct_*` 或 `cf_*` 的查询
2. 检查是否经过 `DocLocalCache` / `batchLocalCacheFirst` / 本地缓存层
3. 评估是否适合缓存：数据量小、变动频率低、多请求间共享 → 适合
4. 适合缓存但未使用的，记录：**表名、查询方法、所在文件:行号**

### 2.5 检查主键值一致的多表查询

**规则**：同一业务主键值关联的多张一对一表，应合并为一条 SQL 一次查询，避免分开逐表查询造成重复 DB 往返。

检查步骤：
1. 在代码中识别同一方法内对多张不同表的查询
2. 判断这些表是否通过**同一主键值**关联（如 `patientId`、`visitId`、`orderId`）
3. 确认是否可以合并为一条 SQL（JOIN 或 UNION ALL）

报告中标明：
| 主键 | 涉及表 | 查询次数 | 查询方法 | 文件:行号 | 是否可合并 |
|------|--------|----------|----------|-----------|-----------|
| patientId | tb_a, tb_b, tb_c | 3次 | findXxx | XxxMapper.java:123 | 是 |

报告中标明：
| 表名 | 表类型 | 查询方法 | 文件:行号 | 是否使用缓存 |
|------|--------|----------|-----------|-------------|
| ct_xxx | 码表 | findXxx | XxxMapper.java:123 | 否 |

## 第三步：日志分析

### 3.1 分页下载日志

使用 graylog-search.md 的 API 格式，以 traceId 为关键字查询日志，limit=500，逐页下载直到返回条数 < 500。

> 对话模式下不下载全量日志，仅通过少量 API 采样确认关键指标。

### 3.2 提取关键指标

用 Node.js 解析所有页面，提取：
- 总耗时 = 最后一条时间戳 - 第一条时间戳
- 日志总数
- 按 `app_name` 分布（确认走了哪些服务）
- 按 `logger_name` 分布（识别高频 Mapper/Service）

### 3.3 识别关键时间点

在日志中搜索特征消息定位各阶段起止：
- `接收到接口.*{方法名}` → 请求到达
- `===>>> 接口.*调用完成` → Feign 调用完成
- `StopWatch` → 分段计时
- `logAfterReturning` → 响应返回

### 3.4 统计操作次数

| 统计类型 | 日志特征 |
|----------|----------|
| Feign 调用 | `===>>> 接口.*调用完成` |
| DB SELECT | Mapper 类名 + `Preparing:` |
| DB INSERT | `INSERT INTO` + `Preparing:` |
| DB UPDATE | `UPDATE` + `Preparing:` |
| 缓存查询 | `DocLocalCache` / `batchLocalCacheFirst` |

### 3.5 区分耗时根因：数据量 vs 代码问题

分析每个耗时环节时，必须明确根因是什么性质：

**数据量大导致的耗时**（非代码缺陷）：
- 单次 SQL 查询扫描行数多（数万行以上）
- 业务确实需要处理大量数据（如批量操作、报表导出）
- DB 返回数据量大导致网络传输耗时
- 标注为"业务数据量大"，非代码问题

**代码缺陷导致的耗时**（可优化）：
- N+1 逐条查询（循环内调 Mapper/Feign）
- 缺少本地缓存（重复查同一数据）
- 主键一致的多表分开查询（可合并）
- 不必要的重计算或重复逻辑
- 标注为"代码缺陷"

### 3.6 量化判断标准

| 判断维度 | 代码缺陷 | 数据量大（非代码问题） |
|----------|----------|----------------------|
| **耗时占比** | 单类操作占总耗时 > 30% | 各项耗时分布均匀，无明显单项集中 |
| **重复模式** | 同一 SQL/Feign 在循环内出现 ≥ 3 次 | 每种查询仅出现 1-2 次 |
| **单次操作耗时** | DB 单条 < 50ms、Feign < 100ms（合理范围） | DB 单条 > 200ms（扫描行数 > 万级） |
| **N+1 特征** | 调用次数 ≈ 数据条数，且单次调用耗时很小 | 批量查询，一次返回大量数据 |
| **缓存缺失** | 同一 ct_/cf_ 表在同次请求中被查 ≥ 3 次 | — |

**核心判断逻辑**：耗时集中在循环内的重复小操作 → **代码缺陷**；耗时集中在单次大数据量查询 → **非代码问题**。

报告中对两类问题分列，不可混淆。数据量问题需标注"非代码问题，需从业务/架构层面解决"。

### 3.7 关键诊断规则

**traceId 查询先看总量、确认后再查全量**：用 traceId 过滤时，先用默认 limit 查一次，看 `total_results` 是否超出限制。如果超出，必须提示用户"共 X 条日志，当前只取了 Y 条，是否查询全部？"，由用户确认后再用 `limit: 1000` 查全量。**禁止在未确认的情况下截取部分数据做分析**，否则会导致耗时归因严重错位。

**按线程+时间戳精确分解**：拿到全量日志后，必须按 `thread_name` 分组、按毫秒级时间戳排序，精确计算每个阶段的耗时，不能凭 StopWatch 总耗时就把子调用归因到一个组件上。

**全线程零日志空白期要标注**：如果某个时间段所有线程都无日志，标注为"不可归因空白期"，不能跳过或忽略。

**问题标题标注产品组**：每个问题标题格式为 `问题 N：{问题描述}【{产品组}】`，只标注最主要负责的那个产品组，不罗列所有相关方。产品组映射参照 [[application-mapping.md]]。

## 第四步：精确排名

### 4.1 估算公式

```
估算总耗时 = 单次平均耗时 × 调用次数
```

单次耗时获取方式（按优先级）：
1. 日志中 `<== Total: N` 的返回值（MyBatis 耗时）
2. `===>>> 接口.*调用完成，耗时：N ms` 的耗时值（Feign 耗时）
3. 根据操作类型估算（DB 单条 ~5-30ms, Feign ~15-50ms）

### 4.2 排名表

```markdown
| 排名 | 操作 | 单次耗时 | 调用次数 | 估算总耗时 | 占比 |
|------|------|----------|----------|------------|------|
| 1 | ... | ~Xms | ×N | **X.Xs** | XX% |
```

占比 = 单项估算耗时 / 所有项估算耗时之和 × 100%

### 4.3 瀑布图

按时间轴展示各阶段耗时和阶段内主要操作。

## 第五步：输出

**对话模式（默认）**：展示基本信息、Top 10 耗时排名、核心问题分析（代码位置+代码片段+影响+优化方向）、优化汇总表。

**文件模式**：当用户明确要求输出文件时，按 [[report-template.md]] 生成完整 MD 报告。

对话模式下不下载全量日志，仅通过少量 API 采样确认关键指标。

## 模式区分

| 用户说 | 执行 |
|--------|------|
| "输出报告" / "生成报告" / "输出文档" | 汇总报告 — 仅对搜索结果整理归类，不包含代码走读、调用链路、N+1 分析、优化方案 |
| "分析 XxxController.xxx" / "分析 traceId xxx" | 深入诊断 — 走完整流程：收集信息 → 读取代码 → 日志分析 → 精确排名 → 输出详细报告 |

汇总报告结构参照 [[report-template.md#报告结构]]，输出到个人配置的分析文档目录下的 `{问题产品组}\`。
深入诊断报告按 [[report-template.md]] 完整模板生成。

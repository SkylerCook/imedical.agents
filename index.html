我的 AI Coding 实践理解

AI Coding 外骨骼架构

AI Coding 不是一个聊天框，而是一套"大脑 + 手脚 + 接口 + 技能 + 规则 + 项目手册 + 多 Agent 协作"的工程化系统。 核心不是只让模型更聪明，而是让模型能在正确上下文、工具、流程和边界里可靠做事。

模型与执行环境解耦 项目上下文对齐 能力可复用 边界可控制
脑
大模型负责想：理解、推理、规划、决策。
手
Codex / Claude Code 客户端负责干：读写文件、运行命令、调用工具、反馈结果。
接口
MCP让手脚能摸到数据库、API、SFTP、IRIS 等外部系统。
手册
Skills / Rules / AGENTS.md让执行过程有 SOP、有红线、有项目共识。
编排
Agents / Workflows让多个专家协同：角色分工、阶段交接、串行降级。
Concept Model

我如何拆解 AI 实践

这套视角先把"大脑"和"手脚"拆开，再给手脚接上工具、技能、规则和项目手册。

1

大模型：大脑

纯粹的认知引擎，负责理解意图、逻辑推理、任务规划和决策。它会想，但不能直接改变真实代码库。

2

客户端：手脚与躯干

在"剥离模型"的视角下，Codex / Claude Code 客户端是执行框架，负责把文本指令变成文件、命令和工具调用。

3

MCP：万能接口

它解决"怎么连接"的问题，让手脚能通过标准协议触达数据库、第三方服务、文件传输和专业系统。

4

Skills：操作手册

它解决"怎么做得专业"的问题，把排障、编码、国际化、测试等专家流程沉淀为可复用 SOP。

5

Rules：安全手套

它解决"什么不能做、必须怎么做"的问题，为 AI 划定代码风格、操作边界和安全红线。

6

AGENTS.md：项目圣经

它把项目背景、全局规则、导航入口和智能体路由打包在一起，让大脑和手脚瞬间知道"我是谁、我在哪、我要遵守什么规矩、谁来做什么"。

Clarification

关键纠偏：我说的是"无模型客户端"

如果把 Codex / Claude Code 看成包含模型的完整产品，它们像 Agent；但如果剥离模型，只看客户端执行框架，它们就是手脚、躯干和外骨骼。

完整产品视角

大模型 + 客户端执行能力 + 工具调用能力组合在一起，可以被称为一个完整 Agent。

>

我的架构视角

把模型拿掉后，客户端就是执行环境：负责读写文件、运行终端、调用工具，并把结果反馈给大脑。

大模型负责想；Codex / Claude Code 客户端负责干；MCP 让手摸到更多工具；Skills 教手怎么干得漂亮；Rules 保证手别干坏事。
Engineering Mapping

从概念到工程实体

《AI Coding Workspace Kit》把这个视角落成了可部署、可复用、可维护的工作区结构。

LLM 大脑

读取项目入口和规则索引，形成任务计划。

AGENTS.md 项目圣经

工程级唯一主入口，提供项目大图、禁区、导航和全局约定。

Agent 客户端

执行文件修改、命令运行、脚本调用和结果反馈。

.agents/agents

智能体注册层，定义角色、职责、权限和交接协议。

.agents/workflows

协作编排层，定义阶段顺序、交接产物和串行降级策略。

交接协议

结构化产物：事实报告、分类清单、变更摘要、验证报告。

.agents/rules

项目规则或插件规则的浅层入口，约束行为边界。

.agents/plugins

可迁移能力包，承载 rules、skills、templates、scripts 和 references。

.agents/skills

真实 skill 或 thin-index，指向插件内完整技能文件。

.mcp.json

MCP、SFTP、IRIS 等连接事实来源；敏感连接信息只放目标工程本地。

thin-index

只暴露轻量入口，需要时再跳转真实文件，避免上下文过载。

.agents/memory

长期记忆，记录决策、坑点、待办和上下文状态。

imedical.agents

这个仓库在做什么

imedical.agents 不是业务系统本身，而是面向医疗软件开发的 AI Coding 能力包与工作区套件。

业务项目部署后的典型结构
your-project/
|-- AGENTS.md
|-- .mcp.json
|-- .agents/
|   |-- agents/            # 智能体注册层
|   |-- workflows/         # 协作编排层
|   |-- config/
|   |-- memory/
|   |-- rules/
|   |-- skills/
|   |-- scripts/
|   `-- plugins/
|       |-- agent-context-kit/
|       |-- coding-iris-plugin/
|       `-- i18n-iris-plugin/
`-- src/
能力包仓库 当前仓库沉淀通用规则、技能、插件、模板和脚本；部署到业务项目后才形成目标工程的 .agents/ 工作区。
连接边界 .mcp.json 是连接事实来源；服务器、账号、密码、token、namespace、远程路径不写入插件。
插件模式 默认使用 plugin-reference-thin-index：插件保留真实文件，浅层入口只做索引。
上下文模式 完整工程使用 codebase-complete；代码不完整或按需导出时使用 intent-first-on-demand-export。
智能体与编排 agents/ 定义角色边界，workflows/ 定义协作边界，plugins/ 定义能力边界。三层分离，串行降级可选。
Capability Packs

插件让 AI 拿到专用工具

没有这些能力包，AI 像拿着通用工具的野战医生；有了这些能力包，AI 更像懂 imedical 系统的专业外科医生。

agent-context-kit

负责初始化和维护项目上下文：生成或合并 AGENTS.md，维护项目 profile、rules、memory，并生成上下文入口。

coding-iris-plugin

面向 IRIS / ObjectScript / CSP / JavaScript / HISUI 的编码能力包，约束本地优先、编译、上传和 GB2312 转换流程。

i18n-iris-plugin

面向医疗系统国际化的专项能力包，覆盖文本提取、翻译 SQL、CSP 翻译同步、XML 模板翻译和质量规则。

Execution Loop

一次任务如何闭环

真正的 AI Coding 不是"模型直接写代码"，而是每次都经过上下文、规则、技能、工具和验证的闭环。

1

需求输入

用户提出目标、范围和约束。

2

读项目圣经

通过 AGENTS.md 对齐项目身份、禁区和导航。

3

加载规则技能

从 agents/workflows 路由、rules、skills 或 thin-index 跳转到插件真实文件。

4

调用工具

通过 MCP 或 scripts 访问外部系统、运行检查、处理文件。

5

验证结果

运行测试、构建、静态检查或人工确认流程。

6

沉淀记忆

把长期有效的决策、坑点和上下文写入 memory；如修正了框架文件，自动生成反馈条目。

Multi-Agent

从单兵作战到专家团队

单 Agent 是一个"大脑 + 手脚 + 手册"的执行体；多 Agent 是多个专科执行体，通过结构化产物交接协作，弱工具自动降级为单 Agent 串行。

怎么协作

角色边界在 agents/，协作流程在 workflows/，能力实现在 plugins/。 阶段之间用事实报告、分类清单、变更摘要和验证报告交接，不共享一大段自由上下文。

探
Explorer查资料、定位链路、确认入口和影响面，输出事实报告。
码
Coder基于 coding / i18n skills 按清单改造代码，输出变更摘要。
审
审查 Agent检查规则、安全边界、残留和代码质量，输出审查报告。
测
测试 Agent运行验证、复现问题、整理失败证据，输出验证报告。
Agent in Practice

从样板到扩展

以 i18n-agent 为例展示实际落地；接入新 Agent 只需一个目录、两个文件、一次 registry 更新。

i18n-agent 实际结构
.agents/agents/i18n-agent/
|-- AGENT.md              # 角色、职责、禁止事项
`-- bindings.yaml         # 插件、规则、skill 绑定

.agents/workflows/
|-- workflow-registry.md
`-- i18n-change.workflow.md
已有样板 AGENT.md 定义角色和禁止事项，bindings.yaml 绑定插件和规则，workflow 定义阶段顺序和交接产物。
接入新 Agent 创建 agents/<name>-agent/，写 AGENT.md + bindings.yaml，选或建 workflow，更新 agent-registry 和 workflow-registry。
adapter 自动生成 从 canonical 生成 .claude/agents/、.codex/agents/、.agents/skills/ 等工具入口；adapter 可删除重建，不维护第二份规则。
能力沉淀为插件 规则、模板、脚本稳定复用时，从 agent 抽离为 plugins/<name>/；agent 只声明可调用能力，不复制插件规则。

一句话总结

未来 AI 应用开发，本质是为大模型这个大脑打造一套可控、可扩展、可复用的 工程外骨骼：用 MCP 连接工具，用 Skills 沉淀经验，用 Rules 划定边界， 用 AGENTS.md 建立项目共识，用 Agents / Workflows 实现多专家协作与阶段化编排。

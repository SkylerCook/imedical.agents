第 1 章

问题引入

医疗信息化场景下，AI Agent 能力碎片化的真实困境

场景

一个熟悉的场景

一个 HIS 需求下来，开发者用 Claude Code 写了一版规则，换到 Codex 又要重新配一遍；好不容易配好了，DeepSeek 模型跑不动，GLM-5 又理解错了……

这不是个别现象，而是行业普遍痛点。

痛点

痛点 1：散落的 Agent 配置

.claude/

Claude Code 专属配置目录，包含 rules、memory 等

.codex/

OpenAI Codex 适配目录，需单独维护一份规则

.codebuddy/

CodeBuddy / WorkBuddy 工具配置，又一套独立体系

各自为政 → 规则重复、版本不一致、经验无法复用

痛点

痛点 2：模型能力偏差

GPT / Claude

指令遵循率 ~90%，复杂规则理解优秀

强模型

DeepSeek

指令遵循率 ~70%，部分复杂规则需简化

中等模型

GLM-5

指令遵循率 ~60%，需结构化引导

中等模型

MiniMax

指令遵循率 ~50%，需大幅降级适配

弱模型

同一套规则，不同模型理解能力差异巨大

第 2 章

解决方案

三层架构 + 四个关键机制，统一 Agent 能力包

架构

三层架构

Canonical

规范定义层

厂商无关的 agent、rule、skill、workflow 定义，一处维护

▼
Capability

能力实现层

插件化的技能、规则、模板，按领域组织，可独立复用

▼
Adapter

工具适配层

按目标工具（Claude Code、Codex、CodeBuddy）自动生成配置

实例

插件目录结构

以 i18n-iris-plugin 为例，展示一个插件的真实组成：

plugins/i18n-iris-plugin/
├── AGENTS.md          # 插件入口（说明书）
├── README.md          # 用途说明
├── manifest.json      # 元数据和依赖声明
├── rules/             # 编码规则（红线）
│   ├── i18n_coding_frontend.md
│   ├── i18n_coding_backend.md
│   └── i18n_verify.md
├── skills/            # 任务流程（SOP）
│   ├── i18n-coding/
│   ├── i18n-text-extract/
│   └── i18n-page-trans-seed/
└── scripts/           # 自动化脚本（工具）
    └── sync-xml-print-template.ps1

📄 AGENTS.md = 说明书

告诉 Agent 这个插件能做什么、怎么用

🚫 rules/ = 红线

必须遵守的编码约束和安全边界

📋 skills/ = SOP

标准化的任务执行流程

🔧 scripts/ = 工具

自动化脚本，减少手工操作

机制

四个关键机制

01

Canonical vs Adapter

规范定义与工具适配分离，canonical 内容不绑定任何单一厂商或模型

02

Thin-Index

轻量索引机制，业务项目只引用需要的能力包，不引入整个仓库

03

插件状态分流

按插件成熟度（stable / beta / draft）分流，避免不稳定内容影响生产

04

模型降级策略

使用 fast / balanced / strong / deep-reasoning 抽象档位，由 adapter 映射到实际模型

实例

Agent 定义：i18n-agent

这就是 Agent 的"岗位说明书"——明确职责、边界和工作流程：

agents/i18n-agent/AGENT.md（片段）
## 适用任务
- IRIS/CSP/HISUI 页面国际化改造
- 字典翻译、打印模板翻译、XML 模板同步

## 不适用任务
- 非 IRIS 平台的国际化
- 数据库迁移、性能优化

## 五阶段模型
| 阶段       | 职责           | 输出物         |
|-----------|---------------|---------------|
| Explorer  | 定位实际链路    | 事实报告       |
| Classifier| 分类用户可见文本 | 分类清单       |
| Coder     | 执行编码改造    | 变更摘要       |
| Template  | 生成翻译种子    | 模板/种子文件   |
| Verifier  | 验证完整性      | 验证报告       |

每个 Agent 都有清晰的职责边界和阶段化输出

实例

Workflow：i18n-change 阶段定义

这就是多 Agent 协作的"流程图"——每个阶段有明确的输入、执行和输出：

workflows/i18n-change.workflow.md（Explorer 阶段）
## Explorer 阶段

**输入**：需求工单编号和描述

**执行步骤**：
1. 读取目标项目 AGENTS.md 确认项目上下文
2. 定位需求涉及的 CSP 页面、JS 文件、后端类
3. 追踪数据链路：前端 → API → 后端 → 数据库
4. 识别所有用户可见的文本来源

**输出**：事实报告（docs/agent-reports/{ticket}/fact-report.md）
- 必须区分：已验证事实 / 推断 / 待确认项
- 不预设数据格式，按实际链路记录

标准化

三大标准化策略

📐

命名约定

agent 目录 kebab-case + -agent，workflow 文件 kebab-case + .workflow.md，统一命名降低认知成本

📂

目录结构

agents/、plugins/、rules/、skills/、workflows/ 职责清晰，每个目录有明确的边界和放置规则

📝

文档驱动

每个插件自带 AGENTS.md、README、manifest，文档即规范，新人可自主上手

实例

一键安装：install-agents.ps1 执行过程

一条命令，5 步自动完成：

第 1 步：克隆仓库

git clone --filter=blob:none --no-checkout ... .agents

延迟下载 + 先不克隆文件，等 sparse checkout 配好再拉取

第 2 步：配置 Sparse Checkout

只拉取：agents/ docs/ workflows/
rules/ skills/ plugins/ scripts/*.ps1

不拉取：AGENTS.md、memory/、README、LICENSE、.github/

第 3 步：配置本地生成层忽略

.agents/.git/info/exclude 忽略：
/config/ /memory/ /rules/
/skills/ /scripts/

防止本地生成物被误提交到能力包仓库

第 4-5 步：校验入口 + 配置忽略

校验 AGENTS.md 必须存在
→ 不存在则报错退出
→ 存在则在 .gitignore 添加 .agents/

AGENTS.md 是唯一主入口，CLAUDE.md/CODEBUDDY.md 只是可选 symlink

复用与扩展

可复用与可扩展

🔌 插件化设计

每个能力域独立为插件，包含 skill、rule、template、script，可单独安装和更新

📋 Thin-Index 机制

轻量索引文件让业务项目按需引用，不引入整个能力包仓库

🧩 新增领域插件

遵循统一模板即可创建新领域插件，如 HIS、LIS、PACS 等医疗子系统

🔧 工具 Adapter

新增 Agent 工具只需编写 adapter 生成逻辑，canonical 内容无需修改

# 需求经验积累机制优化计划

## Context

`feedback/experience/demand-com-exp.md` 是需求处理通用经验文档，当前只有 1 个需求（#6990066）的 4 类经验。随着需求积累，需要解决三个问题：自动总结、文档组织、导航效率。

## 问题分析与方案

### 问题 1：每次需求处理完后自动总结经验

**现状**：无自动化，经验靠手动追加。

**方案**：在 `i18n-coding` 和 `coding-iris-plugin` 的 skill 中增加"需求完成后的经验沉淀"段落，作为阶段化执行的最后一步。

具体做法：
- 在 `i18n-coding/SKILL.md` 的"产出"段落末尾增加一条：需求完成后，检查本次处理中是否有新的通用经验值得沉淀到 `feedback/experience/demand-com-exp.md`。
- 在 `coding-iris-plugin` 的 `iris-backend-coding` 和 `iris-frontend-coding` SKILL.md 中增加类似引导。
- 沉淀触发条件：本次处理中遇到了规则未覆盖的坑、发现了新的判断标准、或某个已有经验命中了新需求号。
- 不强制每次沉淀；没有新经验时不写。

**不做的事**：
- 不新建独立 skill（当前经验积累频率不高，不值得单独 skill）。
- 不用 hook 自动触发（经验沉淀需要人工判断质量，不适合全自动）。

### 问题 2：是否拆分文档

**现状**：4 个章节（后端持久化、前端 DataGrid、需求分析、检验清单），1 个需求号。

**方案：暂不拆分，按领域追加章节。**

理由：
- 当前体量（~110 行）远未到需要拆分的程度。
- 跨领域经验存在（如 3.3 调用链路梳理方法同时适用后端和 i18n），拆分会割裂关联。
- 如果后续 i18n 经验积累到 10+ 条且与非 i18n 经验无交叉，再拆为 `demand-com-exp-i18n.md`。

**组织方式**：
- 按领域追加新章节（如"五、i18n - 打印链路定位"、"六、i18n - 数据分类"）。
- 每条经验用需求号标记来源，支持后续按需求追溯。
- 如果某个领域独立性足够强（如纯 i18n 打印经验），可以在章节标题后标注 `(i18n)` 作为类型标签，方便后续筛选或拆分。

### 问题 3：需求索引和跳转

**现状**：需求号只出现在各条目的"需求"行中，无法快速查看某个需求产出了哪些经验。

**方案：在文档末尾追加"需求索引"章节。**

格式：
```markdown
## 需求索引

| 需求号 | 描述 | 命中经验 |
|---|------|----------|
| #6990066 | 材料字典排序功能 | 1.1, 1.2, 1.3, 1.4, 2.1, 2.2, 3.1, 3.2, 3.3 |
```

- 需求号列使用 markdown 锚点链接，点击可跳转到对应条目。
- 锚点格式：`[需求号](#x-x-标题)`，指向各经验条目的标题。
- 每次新增经验条目时同步更新索引。
- 描述行保持简短（一句话），不复制需求全文。

**锚点生成规则**：
- 每个经验条目的标题（如 `### 1.1 新增字段必须追加到末尾`）自动成为锚点。
- 需求索引中的链接格式：`[#6990066](#11-新增字段必须追加到末尾)`。

## 实施步骤

### 步骤 1：更新 `i18n-coding/SKILL.md` 产出段落

在"产出"段落末尾增加经验沉淀引导。

### 步骤 2：更新 `demand-com-exp.md` 组织方式

- 在维护规则中补充领域标签约定。
- 在文档末尾追加"需求索引"章节，为 #6990066 生成索引条目。
- 为现有经验条目添加锚点链接。

### 步骤 3：更新维护记忆

在 `memory/agent-kit-maintenance-log.md` 中记录变更。

## 验证方式

1. 确认 `i18n-coding/SKILL.md` 产出段落包含经验沉淀引导。
2. 确认 `demand-com-exp.md` 末尾有需求索引，且锚点链接可跳转。
3. 确认维护规则中补充了领域标签约定。

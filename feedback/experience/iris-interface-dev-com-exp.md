# IRIS 接口开发通用经验

> 积累 HIS/IRIS 接口开发、生成、验证和修改过程中的通用检查清单与踩坑记录，按分类组织。

## 维护规则

1. **去重 + 累积**：新增经验前搜索已有条目。若知识点已存在，不重复写内容，只在「需求」行追加新需求号，并 `命中+1`。
2. **需求追溯**：每条经验记录「需求」行，列出命中该经验的所有需求号。跨多个需求沉淀的通用经验标注 `(通用)`。
3. **命中计数**：「命中」数字表示该经验被多少个需求或试运行场景触发过。没有具体需求号时，只递增计数。
4. **分类追加**：新经验按分类追加到对应章节末尾；无法归类时先放入「反馈分流与规则提升」，后续再拆分。
5. **经验粒度**：一条经验对应一个独立知识点，避免混合多个不相关主题。
6. **可操作性**：必须包含具体做法、判断标准、示例命令或落地路径，避免只描述问题。
7. **反馈标记**：当候选经验被提升为 plugin rule、reference、skill、workflow 或 script 时，在本条目追加「已提升: `path/to/file`」标记，不删除原条目。
8. **领域标签**：如果某个经验只适用于特定接口形态，在标题后标注类型标签，例如 `(Query)`、`(JSON)`、`(XML)`、`(Mode B)`。
9. **需求索引**：新增经验条目时，同步更新文档末尾的「需求索引」章节。

### 条目格式

```markdown
### x.x 标题
- 需求: #6990066 #7001234 | 命中: 2
- **规则**：...
- **做法**：...
```

## 二、字段取值与规则匹配

### 2.1 项目私有字段取值不得直接提升为通用规则
- 需求: (通用) | 命中: 1
- **规则**：从具体项目 Global、接口编码、类名前缀、院区配置或本地定制表推导出的字段取值，默认只属于目标项目，不得直接写入通用插件规则库。
- **做法**：先写入目标项目本地 `.agents/rules/his-interface/custom/` 或 `.agents/rules/his-interface/overrides/`；只有经过脱敏、跨项目复用确认和维护者审查后，才能提升到 `his-interface-agent-plugin/rules/`。

### 2.2 L4 人工兜底默认写目标项目 custom/overrides
- 需求: (通用) | 命中: 1
- **规则**：L4 人工指导产生的字段取值规则，默认是项目级知识，不直接写入插件通用 `rules/domains/` 或 `rules/common/`。
- **做法**：先以 dry-run 展示候选规则；用户确认后写入目标项目本地 custom 或 overrides。维护者复核为通用规则后，再从本经验文档追踪提升到插件规则库。

## 六、功能测试与真实数据验证

### 6.1 IRIS 上传、编译和真实数据测试必须由目标项目显式启用
- 需求: (通用) | 命中: 1
- **规则**：上传、编译、远程执行、真实数据查询都可能触达目标环境，不能作为插件默认动作。
- **做法**：离线阶段只生成、审查和报告；当用户明确要求远程验证，且目标项目 `.mcp.json` 或本地私有配置已就绪时，才按项目规则执行 IRIS 上传、编译和功能测试。

## 七、反馈分流与规则提升

### 7.1 框架文件修正走 agent-framework-feedback
- 需求: (通用) | 命中: 1
- **规则**：如果任务中修正了 rules、skills、templates、references、scripts、agents 或 workflows 等框架文件，使用 `skills/agent-framework-feedback` 记录到 `feedback/framework/`。
- **做法**：不要把框架文件 diff 写入本经验文档；本文件只沉淀接口开发通用经验和候选规则提升线索。

### 7.2 通用接口开发经验先进入本文件
- 需求: (通用) | 命中: 1
- **规则**：接口文档解析、字段匹配、代码生成、编译修复、接口定位和验证策略中可跨项目复用的经验，先写入本文件。
- **做法**：条目必须说明触发场景、可操作做法和适用边界；后续若提升为插件 rule/reference/skill，在条目中追加「已提升」标记。

### 7.3 接口索引、MOC、数据流资料并入前必须脱敏审查
- 需求: (通用) | 命中: 1
- **规则**：接口索引、MOC、数据流资料可能包含项目私有类名、接口编码、业务路径或环境线索，并入通用插件前必须审查。
- **做法**：先作为候选参考资料处理；确认不含服务器、账号、token、namespace、远程路径、真实接口敏感事实和项目私有业务逻辑后，才能放入插件 `references/`。

## 八、检查清单

### 8.1 离线试运行优先覆盖只读和 dry-run 链路
- 需求: (通用) | 命中: 1
- **规则**：接口开发插件并入初期，优先验证不触达真实环境的链路，避免试运行阶段产生远程写入或敏感数据访问。
- **做法**：优先执行 `python -m src --help`、`python -m src rule list-domains`、`python -m src rule quality`、`python -m src generate --file <sample> --dry-run` 和 `python -m src review --file <fixture.cls>`；IRIS 编译与真实数据测试放到目标项目显式启用后执行。
